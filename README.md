# SISMID 2026: A Human-AI Teaming Approach to Infectious Disease Modeling

**Building an Influenza Forecasting Pipeline · July 22 through July 24, 2026**

Every week during influenza season, public health decision-makers need to anticipate what is coming: how many people will be hospitalized next week, when the peak will hit, and how high it will go. In this lab you work as a human-AI team to build a national influenza hospitalization forecasting system from real CDC data.

You define the vision and the rules. AI helps build and implement the solution. At the end of the day, you are still the expert making the decisions.

## What you are building

A rules document that evolves into a complete forecasting pipeline and a FluSight-ready model. The pipeline moves through data cleaning, exploration, modeling and forecasting, evaluation, and iterative improvement. Your `rules.md` is the blueprint that describes, in plain language, what the pipeline should do.

Think of AI as a very capable intern: fast and helpful, but it still needs direction. The clearer the guidance, the better the results.

## The human-AI iteration loop

Every activity runs through the same six steps:

1. **Edit the rules.** Open `rules.md` and write or review the rules for that phase.
2. **Generate `agents.md`.** Prompt Copilot to build an agent from your rules document.
3. **Generate the R script.** Use the agent to write the script for this phase.
4. **Run and evaluate.** Execute the script in VS Code. Check the output and results.
5. **Update and regenerate.** Refine the rules, regenerate `agents.md`, regenerate the script.
6. **Push to GitHub.** Automated checks run on your submission.

`rules.md` is written by you and describes **what** the pipeline should do. `agents.md` is generated by AI and describes **how** to execute it. The scripts are what the agent produces.

Vague rules produce vague output. Be precise about every detail.

## Key definitions used throughout

**Influenza season.** MMWR week 40 (roughly early October) through week 20 (roughly mid-May) of the following year. A season spans two calendar years and is named for both, for example 2025-26. Season start is the first calendar date in the first year with epiweek 40. Season end is the earliest calendar date in the second year with epiweek 20. Account for leap years.

**Season attribution note.** When computing `season_start_year` for each `week`, use the calendar year of the date (`year(week)`) as the base year, not the `MMWRyear` returned by `MMWRweek()`. If a date's epiweek is 40 or higher but its calendar month is January through August, attribute it to the previous year's season. This prevents mis-attribution when MMWR weeks cross calendar-year boundaries, and it is worth pasting directly into your rules.

**Reference date and target end date.** A week runs Sunday to Saturday and is labeled with the Saturday. The reference date is the last week of observed data in the training window. Target end dates are the weeks being forecast. Compute the reference date as `reference_date = as_date(t) - days(7)`.

## FluSight submission format

| Column | Type | Acceptable values |
|---|---|---|
| `reference_date` | ISO date | `YYYY-MM-DD` (Saturday) |
| `target` | Character | `wk inc flu hosp` |
| `horizon` | Numeric | `1` |
| `target_end_date` | ISO date | `YYYY-MM-DD` (Saturday) |
| `location` | Character | `US` |
| `output_type` | Character | `quantile` |
| `output_type_id` | Numeric | `0.025`, `0.5`, `0.975` |
| `value` | Numeric | Non-negative integer |

`0.5` is the median point forecast. `0.025` and `0.975` are the lower and upper bounds of the 95% prediction interval. `forecast()` and FluSight describe the same thing in two languages: when you ask for a 95% interval, `forecast()` returns a lower and an upper edge, and those edges are the quantiles FluSight wants.

## Gotchas worth knowing before you hit them

- **Do not double-count the ARIMA variance.** At horizon 1, `pr$se` already incorporates the model's innovation variance. Use `sd_pred <- as.numeric(pr$se)` directly. Adding a separate residual-variance term inflates the interval.
- **`scoringutils` returns `ae_median`, not MAE.** `ae_median` is the absolute error of a single forecast. MAE only exists after you average across reference dates, which is what the by-horizon summary does.
- **95% coverage is not computed by default.** If `score()`'s defaults do not include it, add it explicitly with an interval-coverage metric set to a 95% range.
- **Set `forecast_unit` explicitly** in `as_forecast_quantile()`. Without it, forecasts get silently mis-grouped.
- **Watch for data leakage.** Only data on or before the reference date goes into the model. Write a validation that halts if the training window's maximum week is not strictly less than the target week.
- **Be explicit about dates.** R handles dates unpredictably when left to infer. Specify ISO format on write, and validate that both dates are Date objects and fall on a Saturday.

## Write validations into your rules

Diagnostics are how you confirm the AI interpreted your rules the way you meant them. Useful ones to specify:

- Print the training period start date and the rolling window end dates
- Print the `(p, d, q)` order `auto.arima()` selected at each refit step
- Confirm `target_end_date - reference_date` equals exactly 7 days for every row
- Confirm quantiles are non-decreasing within every target week
- Confirm all values are non-negative integers
- Report how many forecast rows were dropped for lack of an observed target
